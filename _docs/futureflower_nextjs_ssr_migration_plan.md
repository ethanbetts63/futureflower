# FutureFlower Next.js SSR Migration Plan

This document records the initial investigation and migration plan for moving FutureFlower from a Vite React single-page app served by Django to a mostly server-rendered Next.js frontend, deployed separately from the Django API.

Keep this document updated as implementation proceeds. Add dated progress notes at the top after each meaningful migration phase, including build failures that reveal migration constraints.

## Migration progress log

### 2026-05-11 - Investigation and plan created

Starting point:

- Frontend is a Vite React 19 app in `frontend/`.
- Routing is handled by React Router 7 in `frontend/src/App.tsx`.
- Django serves the built SPA from `frontend/dist/index.html` through a catch-all `TemplateView`.
- SEO metadata is currently applied at runtime with `react-helmet-async`.
- Public marketing, article, and order-entry pages are included in the same SPA as authenticated dashboard, partner, admin, checkout, and form-flow routes.
- Django owns `/sitemap.xml`; Vite owns `frontend/public/robots.txt`.
- API calls assume same-origin `/api/...`, with cookie-based JWT auth and CSRF handling.

Important findings:

- The current sitemap includes public routes and also `/login`. `/login` should not be indexable in the Next migration.
- The sitemap lists three article URLs that have page files but are not currently registered in `App.tsx`:
  - `/articles/best-flower-delivery-adelaide`
  - `/articles/best-flower-delivery-darwin`
  - `/articles/best-flower-delivery-melbourne`
- `frontend/public/robots.txt` only disallows `/terms-and-conditions/`. It does not disallow login, checkout, dashboard, partner, admin, password reset, payment, or order-flow routes.
- Tailwind is wired through `@tailwindcss/vite`; Next will need PostCSS/Tailwind setup.
- `import.meta.env.VITE_STRIPE_PUBLISHABLE_KEY` is used in checkout/payment pages and must become a `NEXT_PUBLIC_*` env var.
- `AuthContext`, `NavBar`, `Footer`, protected routes, and most dashboard/flow pages must stay behind client boundaries because they depend on browser state, navigation hooks, auth context, or mutating API calls.

## Executive summary

FutureFlower has more SSR upside than SplitCart because many high-value pages are static or mostly static: homepage, florist partner landing page, affiliate landing page, pricing, contact, occasion landing pages, city landing pages, and article pages. Today, crawlers receive a generic Vite shell first and only see page-specific metadata/content after JavaScript runs.

Recommended target:

1. Create a Next.js App Router frontend in `frontend/`.
2. Keep Django as the API, auth, admin, payment, partner, and data backend.
3. Preserve route parity first with client wrappers.
4. Move SEO metadata, robots, and sitemap ownership to Next.
5. Convert public marketing/article/landing routes to real server-rendered pages.
6. Keep authenticated dashboards, checkout, admin, partner management, and multi-step order flows client-rendered and noindex.

This migration should be less data-fetch-heavy than SplitCart. Most indexable FutureFlower pages already contain their SEO content in React component markup. The first major win is making that markup and metadata available in the initial HTML.

## Recommended target architecture

Target production architecture:

- Next.js frontend deployed on Vercel at `https://www.futureflower.app`.
- Django remains the API/backend application, likely on the current PythonAnywhere host or a dedicated API subdomain.
- Next handles public routes, metadata, sitemap, robots, static assets, and client navigation.
- Django handles `/api/...`, `/admin/`, auth cookies, payments, partner operations, marketing uploads, and background business logic.

Target local development:

- Django: `python manage.py runserver` on port 8000.
- Next: `cd frontend && npm run dev` on port 3000.
- Next rewrites `/api/:path*` to `http://127.0.0.1:8000/api/:path*`.

## API and environment strategy

Add `frontend/next.config.ts` with an API rewrite:

```ts
const nextConfig = {
  async rewrites() {
    const apiUrl = process.env.DJANGO_API_URL ?? "http://127.0.0.1:8000";
    return [{ source: "/api/:path*", destination: `${apiUrl}/api/:path*` }];
  },
};

export default nextConfig;
```

Production environment variables:

- `DJANGO_API_URL` for server-side Next fetches and rewrites.
- `NEXT_PUBLIC_DJANGO_API_URL` if browser API calls need to target an API origin directly.
- `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` to replace `VITE_STRIPE_PUBLISHABLE_KEY`.
- `NEXT_PUBLIC_SITE_URL=https://www.futureflower.app`.

Cookie-auth decision:

- If Django API remains on the same `www.futureflower.app` origin through Next rewrites, the current cookie auth model can mostly remain unchanged.
- If browser calls go directly to another origin such as `api.futureflower.app`, Django cookie settings, CORS, CSRF trusted origins, and credentials behavior must be updated carefully.

Do not mix these accidentally. Authenticated browser flows are the highest-risk part of the deployment.

## Backend deployment changes

Django settings will need updates once the Next frontend is no longer served from `frontend/dist`:

- Remove or reduce reliance on `TemplateView` catch-all for frontend routes.
- Keep `/api/...`, `/admin/`, and backend-only endpoints on Django.
- Decide whether Django still serves `sitemap.xml`; recommendation is Next owns it.
- Update `CSRF_TRUSTED_ORIGINS` for local Next dev and production frontend origin:
  - `http://localhost:3000`

- If using a separate API domain, add that host to `ALLOWED_HOSTS` and configure cookie domain/SameSite/Secure behavior.
- Keep Python/Django staticfiles for Django admin assets and backend templates.

## Migration phases

### Phase 1: Next App Router parity shell

Goal: run the current app under Next with minimal behavior changes.

Tasks:

- Install Next.
- Change scripts:
  - `dev` -> `next dev`
  - `build` -> `next build`
  - add `start`
  - keep/add `typecheck`
- Add `next.config.ts` with API rewrites and path alias support.
- Add `postcss.config.mjs` for Tailwind 4.
- Move global CSS import to `app/layout.tsx`.

- Convert common React Router links/hooks in shared components:
  - `Link` -> `next/link`
  - `useNavigate` -> `useRouter`
  - `useLocation` -> `usePathname` / `useSearchParams`

- `AuthContext` touches `localStorage` and `window`.

- `tw-animate-css` may need a direct CSS import path if Next/Turbopack cannot resolve the package style export.
- Leaflet/react-leaflet components should be dynamically imported with SSR disabled.

Acceptance:

- `npm run typecheck` passes.
- `npm run build` passes.
- Every route currently in `App.tsx` has a Next route or an intentional redirect.
- Public pages render with the same layout.
- Login, logout, order flow, dashboard, checkout, partner, and admin pages still work in local dev.

### Phase 5: Legal terms and dynamic API-backed content

`TermsAndConditionsPage` currently fetches `/api/data/terms/latest/?type=...` client-side and robots disallow terms URLs.

Options:

- Keep terms disallowed and client-rendered. This preserves current policy and is simplest.
- Or server-render terms pages from Django and decide whether they should remain noindex/disallowed.

If server-rendering terms:

- Fetch terms in `app/terms-and-conditions/[type]/page.tsx`.
- Validate `type` against `customer`, `florist`, and `affiliate`.
- Use `notFound()` or redirect invalid types to customer terms.
- Be careful with sanitized HTML and `dangerouslySetInnerHTML`.

## Component conversion notes

### Client-only components/providers

Keep these client-side:

- `AuthProvider`
- `NavBar` initially, because it reads auth context and uses menu state.
- `Footer` initially, because it reads auth context for admin links.
- `Toaster`
- `ProtectedRoute` replacement
- form flow editors
- checkout/payment components
- dashboard and admin components
- partner management pages
- any Leaflet map component

### Good server-render candidates

These look mostly static and should be convertible:

- `ProductCarousel`
- `FaqV2`
- `ArticleCarousel` after replacing `react-router-dom` links
- homepage content sections
- florist/affiliate/pricing content sections
- contact page content sections
- article pages
- occasion/city landing pages

### Images

Initial recommendation:

- Keep plain `<img>` for parity.
- Add a small `assetSrc()` helper if imported images become objects under Next.
- Move to `next/image` after route parity if useful.
- If using `next/image`, configure any remote image domains and be careful with local generated assets.

### Fonts and CSS

Current `index.css` declares local Playfair Display fonts from `src/assets/fonts/...`.

Options:

- Keep the existing `@font-face` CSS initially.
- Later move fonts to `next/font/local` for better preload behavior.

Tailwind 4 must be compiled by Next, not Vite:

- Add `@tailwindcss/postcss`.
- Add `postcss.config.mjs`.
- Test production build CSS output; a tiny CSS bundle usually means Tailwind is not being compiled.

## Backend/API changes worth considering

No backend changes are strictly required for public static SSR, but these would make deployment cleaner:

1. Add explicit CORS/CSRF configuration for the Next dev/prod origins if frontend and API origins differ.
2. Add a stable `SITE_URL`/frontend URL setting used by emails, redirects, and marketing links.
3. Decide whether Django should keep generating sitemap entries or defer fully to Next.
4. Add a lightweight public endpoint for any future dynamic marketing content if landing pages become CMS-backed.
5. Add tests for route redirects and public API availability before changing hosting topology.

## Suggested implementation order

1. Next shell and route parity.
2. Shared navigation/link/router conversion.
3. Tailwind/PostCSS and static asset fixes.
4. Metadata, robots, sitemap.
5. Public marketing pages as SSR/static pages.
6. Article routes and missing sitemap route cleanup.
7. Terms decision.
8. Operational route parity.
9. Remove Vite/React Router/Helmet once unused.
10. Deploy to Vercel with Django API environment configuration.

## Acceptance criteria

For migrated indexable routes:

- Initial HTML contains route-specific title, description, canonical URL, and visible page content.
- Structured data is rendered server-side where appropriate.
- Sitemap contains only indexable routes that return 200.
- Robots/noindex excludes auth, dashboard, checkout, admin, partner operation, password reset, and flow routes.
- Existing order, auth, checkout, dashboard, partner, and admin flows still work after hydration.
- `npm run typecheck` passes.
- `npm run build` passes.
- Local dev works with Django on `127.0.0.1:8000` and Next on `localhost:3000`.

## Open decisions

1. Should `/order` be indexable or treated as noindex conversion UI? Recommendation: noindex initially.
2. Should `/terms-and-conditions/:type` remain disallowed? Recommendation: preserve current disallow initially.
3. Should partner registration pages be SEO pages or operational pages? Recommendation: noindex initially; keep `/florists` and `/affiliates` as acquisition pages.
4. Will Django API remain behind same-origin Next rewrites, or move to a separate API domain?
5. Should missing article routes be added, or should Adelaide/Darwin/Melbourne be removed from the sitemap until ready? Recommendation: add the routes because files already exist.
6. Should `NavBar` and `Footer` be split into server static shells plus small auth-aware client islands later?
